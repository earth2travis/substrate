# The Substrate Spec

A version-controlled shared knowledge graph for the Agent Factory (Zookooree). The Substrate is the nervous system between agents (Koda, Sivart) and Travis (Ξ2T). It follows the Karpathy pattern: raw sources flow through a pipeline into structured knowledge.

## 2. Directory Conventions

### 2.1 `research/raw/` — Capture

Immutable source material. Transcripts, scraped articles, notes, research dumps. Never edited after ingestion. If a source has errors, document them in a finding, don't patch the raw.

### 2.2 `research/findings/` — Process

Auto-generated by `_ingest.py`. Each raw file gets a normalized finding with:
- Consistent frontmatter (title, tags, related, source)
- The original body unchanged
- Tags derived from strict phrase matching
- Related links via wikilink extraction + title mention frequency

Findings are regenerated on each ingest run. If a raw source is deleted, its finding is removed (orphan cleanup).

### 2.3 `research/queries/` — Ask

Stored query definitions. Each `.md` file describes an evaluation question or research prompt that tests or extracts from the Substrate. See **Section 6: Query Spec** for format.

### 2.4 `insights/` — Synthesize

Promoted from findings when a topic appears in 2+ findings. Three subdirectories:

- **`concepts/`** — Ideas, patterns, frameworks. "lean production", "context poisoning pattern", "synthesis vs retrieval paradigm"
- **`entities/`** — People, orgs, systems. "taiichi ohno", "claude code", "hermes agent"
- **`comparisons/`** — Head-to-head analysis. "openclaw vs hermes", "rag vs wiki"

Insights are manually curated or auto-promoted based on cross-reference density. Once promoted, they become stable reference docs and are not overwritten by routine ingests.

### 2.5 `decisions/` — Decide

Architecture Decision Records (ADRs). Every significant structural or policy choice gets a record here.

**Format:**
```markdown
---
title: "Brief decision title"
date: YYYY-MM-DD
status: accepted|deprecated|superseded
supersedes: "[[previous-adrs]]"  # optional
---

# DNN: Title

## Context
What problem or question led to this decision?

## Decision
What did we decide? Be specific.

## Consequences
What are the tradeoffs? What does this enable or constrain?
```

**Numbering:** Files are named `001-decision-title.md`, `002-decision-title.md`, etc., sequentially.

**What belongs here:**
- Structural changes to the repo (adding/removing directories, changing conventions)
- Tool or model selection decisions (why Claude vs GPT for a specific task)
- Policy decisions (what gets promoted to insights, when something is deprecated)
- Architecture choices for Substrate automation (ingestion strategy, linting rules)

**What does NOT belong:**
- Task tracking or TODOs (use GitHub Issues)
- Research findings (go in insights/)
- How-to instructions (go in guides/)

### 2.6 `guides/` — How-To

Practical procedures and workflows. These are instructions for agents or humans to follow.

**Format:**
```markdown
---
title: "Procedure name"
date: YYYY-MM-DD
category: category
---

# Guide: Title

## When to use this
Under what circumstances should this guide be followed?

## Prerequisites
What needs to be set up first?

## Steps
1. Do this
2. Then this
3. Then this

## Verification
How do you know it worked?

## Troubleshooting
What to do if it breaks.
```

**What belongs here:**
- "How to onboard a new agent to the Substrate"
- "How to run the ingest pipeline manually"
- "How to write a proper insight doc"
- "How to create a decision record"
- "How to submit a PR to the Substrate"

**What does NOT belong here:**
- Conceptual knowledge (goes in insights/)
- Raw research (goes in research/)
- Decision rationale (goes in decisions/)

### 2.7 `skills/` — Capabilities (Future)

Intentionally left empty until the foundation is stable. When populated, each subdirectory will document a shared agent capability:

```
skills/
├── substrate-sync/
│   ├── SKILL.md            # Trigger conditions, steps, pitfalls
│   └── scripts/            # Utility scripts for skill execution
├── substrate-ingest/
│   └── SKILL.md
└── substrate-session-summary/
    └── SKILL.md
```

Each skill is a self-contained markdown page with trigger conditions, numbered steps, and known pitfalls. The skill system allows agents to load procedural knowledge on demand rather than holding everything in context.

### 2.8 `evals/` — Evaluate

Evaluation benchmarks and reports. Output from `_query.py` and the Context Evaluation Engine. Each run produces a timestamped report: `evals/YYYY-MM-DD-query-results.md`.

### 2.9 `retros/` — Reflect

Weekly retrospective output. Generated by `_retro.sh` every Sunday, stored as `retros/week-YYYY-WNN.md`.

Retros analyze the Substrate's own health and evolution:

- **What changed:** New raw files, promoted insights, recorded decisions
- **Metrics:** File counts by directory, tag frequency, cross-reference density
- **Patterns:** Emerging themes across the week's ingest
- **Gaps:** Topics with raw but no findings, orphans, stale insights
- **Actions:** What needs attention, what scripts need updates, what specs need revision

Retros are append-only and immutable once written. They feed the continuous improvement loop.

### 2.10 `specs/` — Specifications

Self-describing. Each system component gets a spec doc that defines its purpose, inputs, outputs, and behavior. This is the directory you're reading.

## 3. The Pipeline

Three stages, automated:

```
Capture → Process → Synthesize
raw/   → findings/ → insights/
```

1. **Capture:** Raw sources land in `research/raw/` via agent research, scraping, or manual placement.
2. **Process:** `_ingest.py` normalizes each raw file into `research/findings/` with consistent frontmatter, auto-tags, and cross-references.
3. **Synthesize:** When a concept appears across 2+ findings, it's a candidate for `insights/`. Some promotions are automated (high cross-reference density), others are manual (requiring agent synthesis).

The pipeline runs on a cron schedule (currently 02:30 UTC).

---

## 4. Weekly Retrospectives

Every Sunday, generate a retrospective in `retros/week-YYYY-WNN.md`. This captures:

- **What changed:** New raw files added, insights promoted, decisions recorded
- **Metrics:** File counts by directory, tag frequency analysis, cross-reference density
- **Patterns:** Emerging themes across the week's ingest
- **Gaps:** Topics with raw sources but no findings, orphans, stale insights
- **Actions:** What needs attention, what scripts need updates, what specs need revision

This feed drives continuous improvement. It's not a summary — it's an analysis of the Substrate's own health and evolution.

**Script needed:** `scripts/retro.sh` generates the weekly retrospective from git log, filesystem state, and ingest output.

## 5. Spec: Linting (`_lint.py`)

### 5.1 Purpose

Enforce consistency across all Markdown files in the repo. The linter catches structural problems before they compound in the knowledge graph.

### 5.2 Rules

| Rule | Severity | Check |
|---|---|---|
| `fm-missing` | ERROR | File has no YAML frontmatter block |
| `fm-title-missing` | ERROR | Frontmatter exists but has no `title` field |
| `fm-tags-empty` | WARNING | `tags` field exists but is empty or `[]` |
| `fm-related-missing` | WARNING | No `related` field or fewer than 2 wikilinks |
| `naming-convention` | ERROR | Filename contains underscores, spaces, or uppercase |
| `broken-wikilink` | ERROR | `[[target]]` points to no `.md` file in the repo |
| `orphan-finding` | WARNING | Finding exists but raw source is deleted |
| `no-h1` | INFO | No top-level `# Heading` in the body |
| `duplicate-title` | ERROR | Two files share the same `title` in frontmatter |
| `html-scrape` | ERROR | File contains `<html>` or `<!DOCTYPE` — raw HTML should not be in raw/ |

### 5.3 Input

All `.md` files in the repo (excluding `.git/`).

### 5.4 Output

Structured report to stdout with format:
```
[ERROR] research/raw/some_file.md: fm-title-missing — no title in frontmatter
[WARNING] insights/concepts/foo-bar.md: fm-related-missing — 0 related links
[BROKEN] insights/entities/baz.md: broken-wikilink — [[nonexistent-page]]
```

Exit code 0 if no errors (warnings OK). Exit code 1 if any ERRORs exist.

### 5.5 Auto-fix behavior

The linter should support `--fix` for certain rules:
- `naming-convention`: Rename the file on disk (requires git mv)
- `fm-tags-empty`: Derive tags from strict phrase matching (same logic as ingest)
- `html-scrape`: Move file to a `quarantine/` directory

Non-fixable rules (must be manual): `broken-wikilink`, `duplicate-title`, `fm-missing` (if body is empty).

### 5.6 Usage

```bash
python3 scripts/_lint.py                          # Check all files
python3 scripts/_lint.py --dir research/raw/      # Check specific directory
python3 scripts/_lint.py --fix                    # Auto-fix what it can
python3 scripts/_lint.py --format json            # Machine-readable output
```

### 5.7 Integration

The linter runs:
1. On every PR (pre-merge check)
2. As part of the cron pipeline (before ingest)
3. On demand by agents before committing knowledge files

## 6. Spec: Query (`_query.py`)

### 6.1 Purpose

Execute stored queries against the Substrate to test knowledge coverage, extract synthesized answers, and measure the system's comprehension.

### 6.2 Query File Format

Each query lives in `research/queries/` as a `.md` file:

```markdown
---
title: "Question description"
category: identity|conflict-resolution|temporal-awareness|synthesis
created: YYYY-MM-DD
last_run: YYYY-MM-DD  # updated by _query.py
score: null           # updated by _query.py
---

# Query: Title

## Question
The actual question being asked.

## Expected Answer Elements
- Bullet list of key facts the correct answer should contain
- Sources that should be cited

## Ground Truth
The authoritative answer (written by a human).
```

### 6.3 Query Categories

- **identity:** Who/what is something? Tests entity knowledge.
- **conflict-resolution:** When sources disagree, which wins? Tests provenance handling.
- **temporal-awareness:** What changed and when? Tests temporal reasoning.
- **synthesis:** Cross-domain questions requiring combining multiple insights.

### 6.4 Execution Engine

`_query.py` performs:

1. **Load** all queries from `research/queries/`
2. **Answer** each question using only the files in the Substrate (no external knowledge)
3. **Cite** every file used in the answer
4. **Score** against the ground truth answer
5. **Report** results to `evals/reports/YYYY-MM-DD-query-results.md`

### 6.5 Scoring

Each query is scored 0-100:
- **Entity match (30%):** Did it identify the right entities/concepts?
- **Source citation (20%):** Did it cite the authoritative source?
- **Conflict resolution (25%):** If sources disagree, did it pick the right one?
- **Synthesis depth (25%):** Did it combine multiple sources meaningfully?

### 6.6 Usage

```bash
python3 scripts/_query.py                         # Run all queries
python3 scripts/_query.py --category synthesis    # Run only synthesis queries
python3 scripts/_query.py --single 003-foo.md     # Run one query
python3 scripts/_query.py --format json            # Machine-readable output
```

### 6.7 Integration

Queries run:
1. Weekly as part of the retro process
2. After major ingest runs (to verify knowledge was properly captured)
3. On demand before important decisions requiring Substrate context

## 7. Spec: Ingest (`_ingest.py`)

### 7.1 Purpose

Transform raw, heterogeneous source material into normalized, cross-referenced knowledge files. The ingest pipeline is the heart of the Substrate.

### 7.2 Current Behavior (What _ingest.py Does Today)

The existing `scripts/_ingest.py` already implements:
1. Scan `research/raw/*.md`
2. Parse YAML frontmatter (or infer if missing)
3. Normalize to SCHEMA: title, tags, related, source
4. Auto-tag using strict phrase matching (no single-word keywords)
5. Cross-reference via wikilink extraction + title mention frequency
6. Write normalized findings to `research/findings/`
7. Promote to `insights/` when a topic appears in 2+ findings
8. Orphan cleanup: remove findings/ for deleted raw sources

### 7.3 Required Behavior (What It Should Do)

The current implementation is the baseline. Required additions/changes:

**Must have:**
- Proper handling of `research/queries/`, `decisions/`, `guides/`, `retros/`, `specs/` as separate directories — not touched by ingest
- Explicit logging to a structured log file (currently logs to stdout only)
- Idempotency: running ingest twice produces the same output without duplicates
- Respect `no-ingest` YAML frontmatter flag (skip files marked for manual handling)

**Should have:**
- Incremental mode: only process new/changed raw files since last run
- Tag conflict resolution: when a file manually has tags, don't auto-add conflicting ones
- Related link quality score: weight wikilinks higher than title mentions
- Summary output: what was added, changed, removed since last run

**Nice to have:**
- Content deduplication detection: flag near-duplicate raw files
- Source freshness scoring: prefer newer sources when conflicts exist

### 7.4 Output Format

Ingest produces a summary log:
```json
{
  "run_id": "2026-04-22T03:30:00Z",
  "raw_scanned": 300,
  "html_excluded": 3,
  "findings_written": 297,
  "findings_updated": 12,
  "findings_removed": 2,
  "insights_promoted": 4,
  "insights_already_existed": 15,
  "orphans_cleaned": 2,
  "new_raw_since_last_run": 5,
  "errors": []
}
```

### 7.5 Usage

```bash
python3 scripts/_ingest.py                          # Full run
python3 scripts/_ingest.py --repo /path/to/substrate  # Custom repo path
python3 scripts/_ingest.py --dry-run                # Preview without writing
python3 scripts/_ingest.py --incremental            # Only new/changed files
python3 scripts/_ingest.py --log-file /path/to/log  # Structured output
```

## 8. Spec: Security Scan (`_security-scan.py`)

### 8.1 Purpose

Detect security regressions in the Substrate — primarily around secrets, unsafe tooling patterns, and policy violations. The Substrate is the shared brain; if it's poisoned, all agents trust the poison.

### 8.2 Checks

| Check | Description |
|---|---|
| `leaked-secret` | Scan for patterns matching API keys, tokens, passwords in markdown files |
| `dangerous-command` | Flag markdown containing patterns like `rm -rf`, `DROP TABLE`, `curl | bash` |
| `external-link-rot` | Check external URLs in source fields for 404s (warning only) |
| `policy-violation` | Flag files not following SCHEMA.md conventions (redundant with linter) |
| `stale-source` | Flag findings whose raw source was modified more than N days ago |
| `circular-reference` | Detect wikilink cycles (A→B→C→A) that waste context budget |
| `overlinked` | Files with >10 related links (likely spammy auto-linking) |

### 8.3 Secret Detection Patterns

```python
SECRET_PATTERNS = [
    r'(?i)(api[_-]?key|token|password|secret)\s*[:=]\s*["\']?[a-zA-Z0-9]{16,}',
    r'ghp_[a-zA-Z0-9]{36,}',          # GitHub PAT
    r'sk-[a-zA-Z0-9]{20,}',           # OpenAI/DashScope keys
    r'(?i)bearer\s+[a-zA-Z0-9\-._~+/]+=*',  # Bearer tokens
    r'BEGIN\s+(RSA|EC|DSA|OPENSSH)\s+PRIVATE KEY',  # Private keys
]
```

### 8.4 Output

```
[CRITICAL] research/raw/some-file.md: leaked-secret — found pattern matching GitHub PAT
[WARNING] insights/concepts/foo.md: overlinked — 14 related links (max 10)
[WARNING] research/findings/bar.md: stale-source — raw source last modified 45 days ago
[INFO] No circular references detected
```

Exit code 0 if no CRITICALs. Exit code 1 if any CRITICAL exists.

### 8.5 Usage

```bash
python3 scripts/_security-scan.py                       # Run all checks
python3 scripts/_security-scan.py --only leaked-secret  # Single check
python3 scripts/_security-scan.py --format json          # Machine-readable
```

### 8.6 Integration

Security scan runs:
1. On every PR merge
2. Daily as a cron job
3. Before any automated promotion to insights/

## 9. The Continuous Improvement Loop

```
Ingest → Lint → Query → Retro → Fix
   ↑                          ↓
   └──────────────────────────┘
```

1. **Ingest** processes new raw material
2. **Lint** catches structural problems in the output
3. **Query** tests whether knowledge is actually accessible
4. **Retro** analyzes the week's patterns and identifies gaps
5. **Fix** addresses gaps: update scripts, revise specs, promote insights

This loop is the SEPL loop mentioned in `specs/context-eval-engine.md`. Each iteration makes the Substrate more reliable as a knowledge system.

### 9.1 Cron Schedule

| Job | Schedule | Script |
|---|---|---|
| Ingest | Daily 02:30 UTC | `_ingest.py` |
| Lint | Daily 02:00 UTC (before ingest) | `_lint.py` |
| Security scan | Daily 03:00 UTC (after ingest) | `_security-scan.py` |
| Query evaluation | Weekly Sunday 04:00 UTC | `_query.py` |
| Retro generation | Weekly Sunday 05:00 UTC | `retro.sh` |

## 10. Naming Conventions

All derived from `SCHEMA.md` but enforced by the linter:

- **Files:** kebab-case, lowercase only. `my-topic.md`, never `My_Topic.md`
- **Folders:** lowercase, plural. `insights/`, not `Insight/`
- **Wikilinks:** `[[wikilinks]]` for all internal references. Minimum 2 outbound links per page.
- **Tags:** Broad, stable, no synonyms. One standard tag per concept.
- **Dates:** YYYY-MM-DD format in filenames and frontmatter.

## 11. Relationship to Other Systems

### 11.1 LLM Wiki (`~/wiki/`)

The LLM Wiki is a separate knowledge base with its own ingest cron. The Substrate is the Agent Factory's private nervous system. They're parallel systems with different purposes:

- **LLM Wiki:** Public-facing, curated, about AI/LLM knowledge broadly
- **Substrate:** Private, operational, about the Agent Factory specifically

Content can flow from Substrate to Wiki when a concept is general enough, but never the reverse. The Substrate is the source of truth for team knowledge.

### 11.2 Messaging

Daily status and ephemeral updates go through Telegram/Discord. Only material worth retaining enters the Substrate. Weekly retros summarize messaging activity into structural insights.

### 11.3 GitHub

The Substrate lives on GitHub at `github.com/earth2travis/substrate`. All changes go through PRs. The repo is the version control layer for institutional memory.
